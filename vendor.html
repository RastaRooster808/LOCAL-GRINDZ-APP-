Developer Intake

      

        How a vendor update goes from form submission to live on the app:
      

      
    
        
  1. 
          Vendor submits — fills out this form, copies the generated text, and sends it by text, email, or GitHub issue.
        
    2. 
        
  2. 
          Developer reviews — reads the update and decides if any clarification is needed before making changes.
        
    3. 
        
  3. 
          Developer edits JSON — opens the correct file in /data/ and updates the relevant fields. Menu changes go in menu.json, location changes in locations.json, etc.
        
    4. 
        
  4. 
          Developer logs it — adds an entry to updates.json with status "completed" so there's a record of all changes.
        
    5. 
        
  5. 
          Push to GitHub — commits the changes and pushes to the main branch. GitHub Pages deploys automatically, usually within 1–2 minutes.
        
    6. 
        
  6. 
          Confirm with vendor — lets the vendor know the update is live and asks them to check the app.
        
    7. 
      


      

        Future upgrade path: This static system can be upgraded to
        Firebase Realtime Database, Supabase, Airtable, or GitHub Issues API so vendors
        can submit updates that deploy automatically — no developer in the loop needed.
      

    


    
    

      
Static Site Notes

      

        Because Local Grindz runs on GitHub Pages (no server), there are some limitations to know:
      

      
    
        
  • This form does not save or send data automatically — the vendor must copy and send the text manually.
    • 
        
  • There is no login or authentication — this page is public, so avoid putting sensitive info in the form.
    • 
        
  • Photos cannot be uploaded directly — they must be sent to the developer separately.
    • 
        
  • Changes only go live after the developer edits the JSON files and pushes to GitHub.
    • 
        
  • All data is read-only from the app — the JSON files are the source of truth.
    • 
        
  • The service worker caches files for offline use — after an update, vendors and customers may need to refresh twice to see the latest.
